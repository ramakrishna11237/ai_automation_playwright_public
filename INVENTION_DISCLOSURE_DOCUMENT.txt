SECTION 1 — FIELD OF THE INVENTION
=======================================================================

This invention relates to automated software testing systems, and more
specifically to a method and system for automatically recovering from
UI element locator failures during test execution using a hierarchical
multi-layer recovery engine combined with a persistent, confidence-scored
locator learning database and an assertion-guarded self-healing mechanism.

=======================================================================
SECTION 2 — BACKGROUND / PROBLEM BEING SOLVED
=======================================================================

Modern web applications change frequently. When a developer renames a
button, moves a form field, or updates a CSS class, automated UI tests
break — even though the application itself is working correctly.

EXISTING SOLUTIONS AND THEIR LIMITATIONS:

1. Selenium / WebDriver (traditional)
   - Uses CSS selectors and XPath locators
   - When UI changes, ALL tests using that locator fail
   - No automatic recovery — requires manual locator updates
   - No learning — same failure repeats every run

2. Playwright (current state of the art)
   - Provides semantic locators (getByRole, getByLabel)
   - Still fails when element names change
   - No fallback strategy — single locator, single failure
   - No memory of previous fixes

3. Healenium (existing self-healing tool)
   - Attempts to heal broken locators using tree comparison
   - Does NOT distinguish between action steps and assertion steps
   - Can heal an assertVisible step to a button — causing FALSE PASSES
   - No confidence scoring on healed locators
   - No persistent learning across test runs

4. Testim / Mabl / Applitools (AI-based tools)
   - Cloud-based, require sending test data to third-party servers
   - Black-box — no control over healing logic
   - Expensive subscription model
   - Cannot be run fully on-premise

THE CORE PROBLEM:
  No existing solution combines ALL of the following:
  (a) Ordered multi-layer fallback with defined recovery hierarchy
  (b) Assertion-guarded healing that prevents false test passes
  (c) Persistent confidence-scored locator learning database
  (d) Container-scoped DOM capture (targeted, not full-page)
  (e) Natural language step label to action type mapping
  (f) Fully on-premise, no cloud dependency

=======================================================================
SECTION 3 — SUMMARY OF THE INVENTION
=======================================================================

The invention is a software framework for automated UI test execution
that automatically recovers from element locator failures through a
novel 4-layer ordered recovery engine. When a test step fails to find
its target element, the system attempts recovery through four distinct
layers in strict priority order, each with increasing computational cost.

Successfully healed locators are stored in a persistent learning database
with confidence scores and usage counts, enabling zero-cost recovery on
subsequent test runs. The system includes a novel assertion guard that
prevents self-healing from producing false test passes by blocking
healing of verification steps to interactive elements.

4.1 CORE ARCHITECTURE — THE 4-LAYER RECOVERY ENGINE
----------------------------------------------------------------------

The central invention is the 4-Layer Recovery Engine implemented in
Runner.ts. Every test step passes through exactly these layers in order:

LAYER 1: DIRECT EXECUTION (~5ms)
  - The step's original codegen locator is executed directly
  - Uses Playwright's semantic locators (getByRole, getByLabel, etc.)
  - ~95% of steps succeed here in a stable application
  - If successful: records the locator in the learning database
  - If failed: proceeds to Layer 2

LAYER 2: PARALLEL BATCH STRATEGY FALLBACK (~50ms)
  - Generates 15-20 alternative locator strategies from the step metadata
  - Strategies include: getByRole variants, getByLabel, CSS attribute
    selectors ([aria-label], [data-testid], [name], [placeholder]),
    text-based selectors, and semantic role combinations
  - Executes strategies in parallel batches of 3 using Promise.any()
  - First visible element wins — race condition resolved by visibility
  - If a winner is found: executes the action and stores the fix
  - CRITICAL: Assertion steps (assertVisible, assertText, etc.) SKIP
    layers 2, 3, and 4 entirely — a missing element IS the correct
    result for an assertion, not a failure to recover from
  - If failed: proceeds to Layer 3

LAYER 3: LEARNED FIX LOOKUP (~2ms)
  - Queries the in-memory learning database cache
  - Looks up the original locator as a key
  - Priority: exact action match first, then any action match
  - Sorted by: usage count descending, then timestamp descending
  - Zero disk I/O on the hot path (in-memory cache)
  - If a learned fix exists: attempts execution with the stored locator
  - If successful: increments the usage count in the database
  - If failed: proceeds to Layer 4

LAYER 4: DOM CAPTURE + SELF-HEAL (~50ms)
  - Captures a targeted DOM snapshot (see Section 4.3)
  - Builds a candidate list of possible locators from:
    * Author-provided healHint (highest priority)
    * Original codegen locator
    * Element name extracted from locator string
    * Step label text
    * CSS attribute variants (aria-label, title, placeholder, name)
    * Shadow DOM pierce selectors
    * Fuzzy-matched names from the DOM snapshot (Levenshtein distance)
  - Applies the assertion guard (see Section 4.2)
  - Tries each candidate in order
  - If successful: stores the healed locator in the learning database
  - If all candidates fail: throws a structured error with per-layer
    diagnostic information

FLOWCHART:

  Test Step Received
        │
        ▼
  [Self-Contained?] ──YES──► Execute directly ──► Return result
        │NO
        ▼
  [Layer 1: Direct] ──PASS──► Store in learning DB ──► Return PASS
        │FAIL
        ▼
  [Is Assertion?] ──YES──► Skip healing ──► Return FAIL (correct)
        │NO
        ▼
  [Layer 2: Strategy Batch] ──PASS──► Store fix ──► Return PASS
        │FAIL
        ▼
  [Layer 3: Learned Fix] ──PASS──► Increment count ──► Return PASS
        │FAIL
        ▼
  [Layer 4: DOM + Self-Heal] ──PASS──► Store fix ──► Return PASS
        │FAIL
        ▼
  Structured Error (per-layer diagnostics)

4.2 ASSERTION-GUARDED SELF-HEALING (Novel Safety Mechanism)
----------------------------------------------------------------------

This is a critical novel feature not present in any existing tool.

THE PROBLEM IT SOLVES:
  Without this guard, a self-healing system could "fix" a broken
  assertVisible step by finding a button or link with a similar name.
  The test would PASS even though the expected content is not present.
  This produces false positives — the most dangerous type of test failure
  because it gives false confidence that the application is working.

THE MECHANISM:
  The system maintains two sets:

  ASSERT_ACTIONS = {
    assertVisible, assertHidden, assertCount, assertAttribute,
    assertText, assertValue, assertChecked, assertUnchecked,
    assertEnabled, assertDisabled, assertHasClass, assertInViewport,
    assertEditable, assertFocused, assertUrl, assertTitle,
    validation, assertPattern, assertPdf
  }

  ACTION_ELEMENT_PATTERNS = [
    /^getByRole\('button'/,
    /^getByRole\('link'/,
    /^getByRole\('menuitem'/,
    /^getByRole\('tab'/,
    /^button/,
    /^a\[/,
    /button:has-text/,
    /^a:has-text/
  ]

  RULE: If the current action is in ASSERT_ACTIONS AND the candidate
  locator matches any ACTION_ELEMENT_PATTERN, the candidate is BLOCKED.

  This means:
  - assertVisible("Dashboard heading") will NEVER heal to a button
  - assertText("Record saved") will NEVER heal to a link
  - Only content/display elements are valid heal targets for assertions

  Additionally: Assertion steps skip Layers 2, 3, and 4 entirely.
  If the element is not found on Layer 1, the assertion FAILS correctly.

----------------------------------------------------------------------
4.3 CONTAINER-SCOPED DOM CAPTURE (Novel Efficiency Mechanism)
----------------------------------------------------------------------

Traditional self-healing tools capture the entire page DOM — often
500-2000 elements — and search through all of them. This is slow and
produces many false matches from unrelated page sections.

THE NOVEL APPROACH — 3-LEVEL TARGETED CAPTURE:

  Level 1: Nearest Container Snapshot
    - Walks up the DOM from the failing locator
    - Finds the nearest meaningful ancestor:
      [role='dialog'] > [role='alertdialog'] > form > section > main > body
    - Takes an aria snapshot of ONLY that container
    - Typically 5-20 elements instead of 500+
    - Provides self-heal with only the relevant sibling elements

  Level 2: Visible Viewport Snapshot
    - If Level 1 fails, captures only elements visible in the viewport
    - Uses page.evaluate() to get viewport bounds
    - Finds the most specific visible container within those bounds
    - Ignores off-screen modals, hidden menus, collapsed sections

  Level 3: Full Page Aria Snapshot
    - Last resort — full accessibility tree snapshot
    - Still faster than raw HTML because aria snapshots are pre-filtered

  Level 4: Filtered HTML (absolute last resort)
    - Strips scripts, styles, and non-interactive elements
    - Applies SmartDOMFilter to extract only interactive elements

  BENEFIT: Level 1 succeeds in ~90% of cases, reducing DOM processing
  from O(n=500) to O(n=15) — a 33x reduction in search space.

4.4 PERSISTENT CONFIDENCE-SCORED LOCATOR LEARNING DATABASE
----------------------------------------------------------------------

The learning database (learning-db.json) stores healed locator mappings
with the following schema:

  {
    "old":       string,   // original failing locator or step label
    "new":       string,   // healed locator that succeeded
    "action":    string,   // action type (click, fill, assertVisible...)
    "label":     string,   // human-readable step label
    "timestamp": number,   // Unix timestamp of last successful use
    "success":   boolean,  // whether this fix has been confirmed
    "count":     number    // number of times this fix has been used
  }

NOVEL FEATURES OF THE LEARNING SYSTEM:

  1. IN-MEMORY CACHE: The database is loaded once per process into
     memory. All reads are O(1) cache hits — zero disk I/O on the
     hot path. Cache is invalidated only on write.

  2. CONFIDENCE SCORING: Fixes are ranked by:
     - Exact action match (click fix used for click steps)
     - Usage count (higher count = more reliable)
     - Recency (more recent = more likely still valid)

  3. WRITE THROTTLING: The throttledUpdateFix() function only writes
     to disk when count crosses a threshold (every 10 uses), preventing
     excessive disk I/O during high-frequency test runs.

  4. AUTOMATIC PRUNING: The database is pruned on every write:
     - Entries older than 30 days are removed (TTL)
     - If > 500 entries, lowest-count entries are removed
     - Prevents unbounded database growth

  5. ATOMIC WRITES: Database writes use a tmp file + rename pattern
     to prevent corruption if the process crashes mid-write.

  6. GENERIC LOCATOR GUARD: Generic locators (button, a, input, div)
     are never stored in the learning database — they would match
     too many elements and produce incorrect heals.

----------------------------------------------------------------------
4.5 NATURAL LANGUAGE STEP LABEL TO ACTION TYPE MAPPING
----------------------------------------------------------------------

The PatternEngine detects the intended action from a human-readable
step label, eliminating the need to specify action types explicitly.

EXAMPLES:
  "Login to the application"        → action: "login"
  "Enter username"                  → action: "fill"
  "Click the Save button"           → action: "click"
  "Verify dashboard is visible"     → action: "assertVisible"
  "Wait for network to settle"      → action: "waitForNetwork"
  "Upload the report file"          → action: "upload"
  "Double click the record row"     → action: "doubleClick"
  "Assert URL contains /dashboard"  → action: "assertUrl"

DETECTION ALGORITHM:
  - Ordered keyword matching with specificity rules
  - More specific patterns checked before generic ones
  - Auth patterns checked before generic "enter/type" patterns
  - Assertion patterns checked before generic "verify" patterns
  - Falls back to "click" if no pattern matches

----------------------------------------------------------------------
4.6 FUZZY TEXT MATCHING IN SELF-HEAL
----------------------------------------------------------------------

The self-healing engine uses Levenshtein edit distance to match element
names from the DOM snapshot against the target element name.

This catches common UI text changes such as:
  "Record saved."    → "Record updated."    (edit distance: 4)
  "Save Changes"     → "Save"               (substring match)
  "Submit Form"      → "Submit"             (substring match)

ALGORITHM:
  - Exact match: distance = 0
  - Substring match: one string contains the other
  - Levenshtein: only applied to strings ≤ 50 chars (avoids O(n²))
  - Maximum distance threshold: 3 characters

SECTION 5 — PATENT CLAIMS
=======================================================================

CLAIM 1 — THE 4-LAYER ORDERED RECOVERY ENGINE:

  A computer-implemented method for automated UI test step execution
  comprising:
  (a) attempting direct execution of a test step using an original
      element locator (Layer 1);
  (b) upon failure of step (a), generating a plurality of alternative
      locator strategies and executing them in parallel batches using
      a race-condition resolution mechanism to identify a first
      visible matching element (Layer 2);
  (c) upon failure of step (b), querying a persistent in-memory
      locator learning cache using the original locator as a key
      to retrieve a previously successful locator (Layer 3);
  (d) upon failure of step (c), capturing a targeted DOM snapshot
      scoped to the nearest ancestor container of the failing element,
      generating heal candidates from the snapshot, and attempting
      execution with each candidate (Layer 4);
  wherein steps (a) through (d) are executed in strict sequential
  order and each layer is attempted only upon failure of the
  preceding layer.

CLAIM 2 — ASSERTION-GUARDED SELF-HEALING:

  The method of Claim 1, further comprising:
  an assertion guard mechanism that:
  (a) maintains a set of assertion action types including assertVisible,
      assertHidden, assertText, assertValue, and related verification
      actions;
  (b) maintains a set of interactive element locator patterns including
      button roles, link roles, and clickable element selectors;
  (c) when the current test step action is an assertion type, skips
      Layers 2, 3, and 4 entirely, causing the step to fail correctly
      when the expected element is not found;
  (d) when evaluating heal candidates for assertion steps, blocks any
      candidate whose locator matches an interactive element pattern;
  whereby false test passes caused by healing assertion steps to
  interactive elements are prevented.

CLAIM 3 — PERSISTENT CONFIDENCE-SCORED LOCATOR LEARNING:

  A system for persistent locator learning in automated UI testing
  comprising:
  (a) a learning database storing records of the form
      {old_locator, new_locator, action, count, timestamp, success};
  (b) an in-memory cache layer that loads the database once per
      process and serves all read operations from memory;
  (c) a confidence scoring algorithm that ranks stored fixes by
      action-type match, usage count, and recency;
  (d) a write throttling mechanism that defers disk writes until
      a usage count threshold is reached;
  (e) an automatic pruning mechanism that removes entries exceeding
      a time-to-live threshold and caps total entries by usage count;
  (f) a generic locator guard that prevents storage of non-specific
      locators that would produce incorrect matches;
  whereby the system improves locator resolution accuracy over time
  through accumulated test execution experience.

CLAIM 4 — CONTAINER-SCOPED DOM CAPTURE:

  A method for targeted DOM capture in automated UI test self-healing
  comprising:
  (a) identifying the failing element's locator as a hint;
  (b) traversing the DOM ancestor hierarchy from the failing element
      to find the nearest meaningful container element selected from
      the ordered set: dialog, alertdialog, form, section, main, body;
  (c) capturing an accessibility tree snapshot scoped to only the
      identified container element;
  (d) upon failure of step (c), capturing an accessibility tree
      snapshot scoped to elements visible within the current viewport;
  (e) upon failure of step (d), capturing a full-page accessibility
      tree snapshot;
  whereby the search space for self-healing candidates is reduced
  from the full page element count to the relevant container element
  count, improving both performance and healing accuracy.

CLAIM 5 — NATURAL LANGUAGE ACTION DETECTION:

  A method for automated test action type inference comprising:
  receiving a test step comprising a human-readable label string
  and optionally an element locator, and determining the action
  type to perform by applying an ordered set of keyword pattern
  rules to the label string, wherein:
  (a) authentication patterns are evaluated before generic input patterns;
  (b) specific interaction patterns are evaluated before generic patterns;
  (c) assertion patterns are evaluated before generic verification patterns;
  (d) navigation patterns are evaluated before generic action patterns;
  whereby test authors can specify test steps using natural language
  labels without explicitly declaring action types.

CLAIM 6 — FUZZY DOM ELEMENT MATCHING:

  The method of Claim 4, wherein generating heal candidates from the
  DOM snapshot further comprises:
  (a) extracting element names from the accessibility tree snapshot;
  (b) computing the Levenshtein edit distance between each extracted
      name and the target element name;
  (c) accepting candidates where the edit distance is within a
      configurable threshold or where one name is a substring of
      the other;
  whereby the system recovers from minor UI text changes such as
  label renames, punctuation changes, and word additions or removals.
